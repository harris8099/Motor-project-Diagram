🔁 2. Data Flow Sequence

Chronological sequence of data transmission from edge telemetry to cloud storage, background ML processing, and UI presentation.

sequenceDiagram
    participant ESP as ESP32 (Edge)
    participant RND as Render.com (FastAPI)
    participant NEO as Neon (PostgreSQL)
    participant WKR as Background Worker
    participant VRC as Vercel (React)
    participant OMA as Open-Meteo API
    Note over ESP: Every 5 seconds
    ESP->>RND: POST /data {SensorPayload}
    RND->>NEO: SELECT last fault state
    RND->>OMA: GET ambient temp (if t2 == 0)
    OMA-->>RND: temperature_2m
    RND->>NEO: INSERT sensor_readings
    RND->>NEO: INSERT/UPDATE fault_logs
    RND->>WKR: enqueue_prediction_job()
    RND-->>ESP: {ok: true}
    Note over WKR: Async ML Pipeline
    WKR->>NEO: SELECT 72h history
    WKR->>WKR: Run ML Pipeline
    WKR->>NEO: INSERT predictions
    Note over VRC: User Dashboard
    VRC->>RND: GET /data/{device_id}
    RND-->>VRC: JSON response
    VRC->>RND: POST /ai/analyze/{id}
    RND-->>VRC: AI commentary

🛡️ 5. Motor Protection Signal Flow

Local safety logic executing on the ESP32 to autonomously shut down the motor during fault conditions.

flowchart TD
    READ["Read Sensor Data"] --> GRACE{"Startup Grace?"}
    GRACE -->|Yes| SUPPRESS["Suppress faults"]
    GRACE -->|No| CHECK["Evaluate Raw Faults"]
    SUPPRESS --> CHECK
    CHECK --> OC{"Current gt max?"}
    CHECK --> OT{"Temp gt max 3x?"}
    CHECK --> ST{"RPM lt min and I gt stall?"}
    CHECK --> VIB{"Vibration gte threshold?"}
    CHECK --> OV{"Voltage gt max?"}
    CHECK --> UV{"Voltage lt min?"}
    OC --> LATCH["Latch Fault Flags portMUX"]
    OT --> LATCH
    ST --> LATCH
    VIB --> LATCH
    OV --> LATCH
    UV --> LATCH
    LATCH --> NVS["Persist to NVS Flash"]
    NVS --> COUNT{"fault_count gte trip?"}
    COUNT -->|Yes| TRIP["Relay OFF - Motor Stopped"]
    COUNT -->|No| CONT["Continue Running"]

🧠 6. ML Prediction Pipeline

Tiered cloud analysis combining classical rule-based checks, Sklearn models, and generative AI fallback chains.

flowchart TD
    IN["Sensor Payload Received"] --> DISP["enqueue_prediction_job"]
    DISP --> MODE{"PREDICTION_MODE"}
    MODE -->|basic| T1["Tier 1 Rule-Based - Overheating / Stall / Bearing"]
    MODE -->|ml| T2["Tier 2 sklearn ML - Isolation Forest, LinReg, Poly"]
    MODE -->|advanced| T4["Tier 4 Advanced ML - IF, OCSVM, RF, NASA BHI, RUL"]
    T1 --> STORE["INSERT INTO predictions"]
    T2 --> STORE
    T4 --> STORE
    STORE --> AI["POST /ai/analyze"]
    AI --> M1{"Mistral AI"}
    M1 -->|Fail| M2{"Google Gemini"}
    M2 -->|Fail| M3{"Groq"}
    M1 --> OUT["Expert Commentary / RUL JSON"]
    M2 --> OUT
    M3 --> OUT

🗄️ 7. Database Schema

Relational entity model hosted on Neon Serverless PostgreSQL.

erDiagram
    devices ||--o{ sensor_readings : "generates"

    devices ||--o{ fault_logs : "has"
    devices ||--o{ commands : "receives"
    devices {
        TEXT id PK
        BOOLEAN is_active
        TIMESTAMPTZ last_seen_at
        TEXT location
    }
    sensor_readings {
        BIGSERIAL id PK
        TEXT device_id FK
        TIMESTAMPTZ ts
        FLOAT voltage
        FLOAT current
        FLOAT temp1
        BOOLEAN fault_overcurrent
    }
    predictions {
        BIGSERIAL id PK
        TEXT prediction_type
        FLOAT confidence
        TEXT severity
    }
    fault_logs {
        SERIAL id PK
        TEXT fault_type
        TEXT status
        TIMESTAMPTZ detected_at
    }
    commands {
        SERIAL id PK
        TEXT command
        TEXT status
    }
